#O que é um blind dead drop?

Um dead drop (ponto morto) clássico é um local onde uma pessoa deixa algo para outra recolher depois, sem que as duas nunca se encontrem. Um dead drop cego (blind) acrescenta uma propriedade: o próprio local — o servidor — não é confiável. Ele guarda o pacote, mas não consegue abri-lo, não sabe quem o deixou nem quem o pegou, e não consegue ligar dois pacotes como pertencentes à mesma troca.

É exatamente esse o modelo de ameaça aqui. O servidor é um armazenamento chave→valor burro e cego (uma única tabela MySQL), acessível por HTTP(S). Os dois clientes compartilham um único segredo por um canal externo e derivam todo o resto a partir dele.

O servidor só enxerga…

  • endereços de 64 hex com cara de aleatórios
  • blobs de bytes cifrados
  • um prazo de expiração grosseiro, em baldes

O servidor nunca enxerga…

  • o segredo compartilhado ou qualquer chave
  • qualquer mensagem em texto claro
  • quais blobs formam uma mesma conversa

#Como funciona

As duas partes compartilham um segredo raiz de 32 bytes. Um slot lógico é um canal (um número) que guarda dois payloads: um request (pedido) e uma response (resposta). A partir do segredo, cada cliente deriva — para um dado canal e parte:

endereço = HKDF(segredo, "bdd-addr|<parte>|<canal>")   → 64 chars hex
chave    = HKDF(segredo, "bdd-key|<parte>|<canal>")    → 32 bytes

O cliente cifra a mensagem com ChaCha20-Poly1305 sob essa chave e faz PUT nesse endereço. Como request e response derivam de rótulos diferentes, eles vão parar em endereços não relacionados, sob chaves não relacionadas — e é por isso que o servidor não consegue conectá-los. O blob no fio é simplesmente:

nonce(12 bytes) ‖ tag(16 bytes) ‖ texto cifrado

A ida e volta

  Parte A (pede)               Servidor (cego)           Parte B (processa)
  ───────────────              ───────────────           ──────────────────
  sela o request, deriva addr
  PUT /v1/slot/<addrR>  ───────▶ guarda o blob
                                                  ◀─────── GET /v1/slot/<addrR>?wait
                                 devolve o blob ──────────▶ abre + processa
                                                  ◀─────── PUT /v1/slot/<addrS>
  GET /v1/slot/<addrS>?wait ────▶ devolve o blob
  abre a response ◀────────────────────────────────────  (o servidor não ligou nada)

Ambos os GET podem fazer long-poll com ?wait=N: o servidor segura a requisição aberta até o slot ser preenchido ou passarem N segundos, então a troca acontece ao vivo, sem ficar consultando repetidamente.

#Início rápido

Localmente, o servidor embutido fala HTTP (o php -S não faz TLS — em produção o TLS fica num proxy reverso ou serviço onion na frente). Precisa de PHP 8.4+ e um MySQL/MariaDB.

# 1. Cria o schema (lê o .env do projeto) e sobe o servidor cego
bin/bdd migrate
bin/bdd serve --port 8080

# 2. Um segredo compartilhado que as duas partes guardam (32 bytes, hex)
export BDD_SECRET=$(bin/bdd keygen)

# 3. Deixe um request e uma response no canal 0
bin/bdd send --part request  --channel 0 --message "ping?" --port 8080
bin/bdd send --part response --channel 0 --message "pong!" --port 8080

# 4. Leia de volta
bin/bdd recv --part request  --channel 0 --port 8080
bin/bdd recv --part response --channel 0 --port 8080

# Ou bloqueie até aparecer uma mensagem (long-poll do servidor)
bin/bdd recv --part response --channel 0 --wait --timeout 60 --port 8080
Para falar com o serviço ao vivo, troque por --host darkgoldenrod-gnat-566022.hostingersite.com --port 443 --scheme https.

#Referência da API

Toda a superfície são quatro rotas. Cada <endereço> tem exatamente 64 caracteres hex minúsculos; qualquer outra coisa é 404.

RotaPara quêRespostas
GET /v1/health Liveness + os baldes de expiração permitidos. 200 JSON
PUT /v1/slot/<endereço>
?ttl=N		 Guarda um blob (escrita única). O ttl em segundos é arredondado para cima até um balde; ausente usa o padrão do servidor. Corpo de 1 byte até 1 MiB. 201 criado
409 ocupado
413 vazio/grande demais
GET /v1/slot/<endereço>
?wait=N		 Busca um blob. wait faz long-poll por até N segundos (teto do servidor: 60). 200 blob
404 vazio/expirado
DELETE /v1/slot/<endereço> Remove um slot (ex.: apagar após ler). 204 apagado
404 ausente

Expiração padronizada

Toda entrada expira. Para não vazar um prazo de vida preciso por mensagem, o ttl pedido é arredondado para cima até o balde permitido mais próximo — 300, 3600, 86400, 604800 (5 min / 1 h / 1 dia / 1 semana). A expiração é aplicada em toda leitura (um slot expirado responde 404 e é apagado na hora). Os baldes são anunciados pelo endpoint de health.

Exemplos de requisição (curl cru)

HOST=https://darkgoldenrod-gnat-566022.hostingersite.com

# Health — a única chamada que não precisa de endereço derivado do segredo
curl -s $HOST/v1/health
# → {"status":"ok","ttl_buckets":[300,3600,86400,604800]}

# Guarda um blob por 1 hora num endereço (derivado pelo cliente)
ADDR=59050b4ec15597c9f30cb39a5ddab76b17de9fb50120cba7e67b904579d98ffe
curl -s -X PUT --data-binary @blob.bin "$HOST/v1/slot/$ADDR?ttl=3600"

# Long-poll nesse slot por até 30 s
curl -s "$HOST/v1/slot/$ADDR?wait=30" -o blob.bin

# Apaga
curl -s -X DELETE "$HOST/v1/slot/$ADDR"
O endereço e o blob não são coisas que você inventa — eles são derivados e cifrados a partir do segredo compartilhado por um cliente. O derivador abaixo mostra exatamente como um endereço é calculado.

#Experimente — derive um endereço de slot

Isto roda inteiramente no seu navegador usando a Web Crypto API, com a mesma derivação HKDF-SHA256 que os clientes (e o servidor PHP) usam. Cole um segredo de 32 bytes (64 caracteres hex), escolha a parte e o canal, e você obtém o endereço exato sob o qual o servidor guardaria. Nada sai da página.

Com um endereço em mãos, a URL da requisição é simplesmente https://<host>/v1/slot/<endereço>. Derivar a chave por mensagem e cifrar o payload exige ChaCha20-Poly1305, o que os clientes de linha de comando fazem por você.

O demo entre linguagens

Um único script roda três RPCs cegas em que quem pede e quem processa são linguagens diferentes. Cada worker espera um request, processa (coloca o texto em maiúsculas — no lugar de um trabalho real) e posta a resposta.

bash examples/demo.sh
== Exchange 1 — Python asks, C++ processes (channel 1) ==
  Python -> request : 'hello from python'
  Python <- response: 'HELLO FROM PYTHON'  (computed by C++)

== Exchange 2 — C++ asks, Python processes (channel 2) ==
  C++ -> request : 'hello from c++'
  C++ <- response: 'HELLO FROM C++'  (computed by Python)

== Exchange 3 — PHP asks, Python processes (channel 3) ==
  PHP -> request : 'hello from php'
  PHP <- response: 'HELLO FROM PHP'  (computed by Python)

PHP (nativo)

Importa Bdd\Client do pacote; criptografia pelos primitivos auditados do PHP (OpenSSL chacha20-poly1305 + hash_hkdf).

Python e C++ (do zero)

Implementam ChaCha20-Poly1305 e HKDF do zero (validados contra os vetores RFC 8439 / 5869) para mostrar que o formato de fio é um contrato entre linguagens.
